# Starter Backend ‚Äî Agent Guide

Hono backend with clean architecture. One codebase, two runtimes: Cloudflare Workers (D1 + KV) and AWS Lambda (in-memory repositories for now).

## Commands

```bash
npm run dev                 # wrangler dev at http://localhost:8787
npm run typecheck           # tsc --noEmit (run this before finishing any change)
npm run build:lambda        # esbuild bundle ‚Üí dist/lambda/index.mjs
npm run db:migrate:local    # apply D1 migrations locally (needed before first dev run)
npm run db:migrate:remote   # apply D1 migrations on Cloudflare
npm run deploy              # wrangler deploy
```

There is no test suite yet. Verify changes with `npm run typecheck` and, for behavior changes, run `npm run dev` and hit the endpoints with curl.

## Architecture

Dependency rule: outer layers depend on inner layers, never the reverse.

```
routers ‚Üí handlers ‚Üí services ‚Üí domain/repositories (interfaces)
                                        ‚Üë implemented by
                                 infrastructure (d1 / kv / memory)
```

- `src/domain/` ‚Äî entities, `AppError` subclasses, repository **interfaces**. No framework or runtime imports allowed here (no `hono`, no `D1Database` in signatures).
- `src/infrastructure/` ‚Äî repository implementations per backing store: `d1/`, `kv/`, `memory/`.
- `src/services/` ‚Äî business logic. Depends only on domain interfaces. Throws `NotFoundError` / `ConflictError` / `ValidationError` from `src/domain/errors.ts`; never returns HTTP responses.
- `src/handlers/` ‚Äî classes with arrow-function methods taking Hono `Context`. Parse input, call service, return `c.json()`. No business logic.
- `src/routers/` ‚Äî route declarations plus OpenAPI metadata (`describeRoute`) and request validation (`validator` from `hono-openapi`). All resource routers are mounted at `/api/v1` in `src/routers/index.ts`.
- `src/schemas/` ‚Äî zod schemas describing the HTTP contract (request bodies, params, response shapes). Used for both validation and OpenAPI generation. Keep in sync with domain entities.
- `src/di/container.ts` ‚Äî wires repositories ‚Üí services ‚Üí handlers. All construction happens here.
- `src/app.ts` ‚Äî runtime-agnostic app factory. Takes a container factory, sets it on context per request, maps `AppError` to HTTP status in `onError`. Also serves the OpenAPI spec at `/openapi.json` (generated by `hono-openapi` from route metadata) and Scalar docs UI at `/docs`.
- `src/server.ts` ‚Äî Cloudflare Workers entry (wired with D1 + KV). Referenced by `wrangler.jsonc` `main`.
- `src/lambda.ts` ‚Äî AWS Lambda entry via `hono/aws-lambda` (wired with memory repos).

## Adding a new resource (e.g. bookings)

Follow the `user` files as the template, in this order:

1. `src/domain/entities/booking.ts`
2. `src/domain/repositories/booking-repository.ts` (interface)
3. `src/infrastructure/d1/d1-booking-repository.ts` (+ memory impl if Lambda needs it)
4. `src/services/booking-service.ts`
5. `src/handlers/booking-handler.ts`
6. `src/schemas/booking-schemas.ts` (zod) and `src/routers/booking-router.ts` with `describeRoute` + `validator`, mounted in `src/routers/index.ts`
7. Wire into `src/di/container.ts` and both entrypoints (`server.ts`, `lambda.ts`)
8. New migration in `migrations/` (numbered, e.g. `0002_create_bookings.sql`), then `npm run db:migrate:local`

## Conventions

- Filenames: kebab-case, prefixed by implementation (`d1-user-repository.ts`, `kv-cache-repository.ts`).
- D1 rows use snake_case columns; map to camelCase entities inside the repository (see `toUser` in `d1-user-repository.ts`).
- Route params: use the `param(c, 'id')` helper pattern in handlers ‚Äî `c.req.param()` returns `string | undefined` on an untyped Context, and `noUncheckedIndexedAccess` is on.
- Errors: throw domain errors from services; `app.onError` converts them to `{ error: { code, message } }` JSON. Don't try/catch in handlers except for JSON parsing.
- Success responses are wrapped as `{ data: ... }`.
- Validation is two-layer: `validator('json'|'param', schema)` in routers rejects malformed requests with 400 before the handler runs; services still enforce business rules (duplicate email, etc.) as the source of truth.
- Zod v4 syntax: `z.email()`, `z.uuid()`, `z.iso.datetime()` (not the deprecated `z.string().email()` chains).

## Gotchas

- `wrangler.jsonc` contains placeholder IDs (`<REPLACE_WITH_...>`) for D1 and KV. Local dev works with placeholders; real deploys need real IDs from `wrangler d1 create` / `wrangler kv namespace create`.
- KV enforces a minimum TTL of 60 seconds ‚Äî `KVCacheRepository.set` clamps for you.
- `@cloudflare/workers-types` must be v5+ to satisfy wrangler v4's peer dependency.
- Lambda has no D1/KV; `src/lambda.ts` uses memory repositories. Swap for DynamoDB/RDS implementations there without touching services.
- CI (`.github/workflows/ci.yml`) deploys to Cloudflare on push to `main` (needs `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` secrets). The Lambda deploy job is commented out.
